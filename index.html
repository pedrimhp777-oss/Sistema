01 — Evolução

As ferramentas ao longo do tempo

Cada geração de VCS surgiu para resolver as limitações da anterior, culminando no Git como padrão global.

1986
CVS — Concurrent Versions System
Primeira ferramenta a ganhar adoção massiva. Permitia que múltiplos desenvolvedores atualizassem um repositório central, mas sofria com ausência de commits atômicos — falhas de rede podiam corromper o repositório inteiro.
Centralizado Obsoleto
2000
SVN — Apache Subversion
Criado para substituir o CVS, introduziu commits atômicos (tudo ou nada), melhor rastreamento de diretórios e metadados. Tornou-se padrão corporativo na primeira década dos anos 2000 e ainda é usado em ambientes empresariais.
Centralizado
2005
Mercurial
Marcou a transição para o modelo distribuído — cada desenvolvedor possui uma cópia completa do repositório. Interface amigável e alta performance. Perdeu terreno para o Git após o GitHub se consolidar como plataforma dominante.
Distribuído
2005
Git ★ Padrão atual
Criado por Linus Torvalds para o desenvolvimento do kernel Linux. Trata dados como snapshots do sistema de arquivos. Branching extremamente rápido e barato. Mais de 93% dos desenvolvedores profissionais usam Git (Stack Overflow Survey 2023).
Distribuído

03 — GitHub

Da implementação ao uso

O GitHub adiciona colaboração remota, hospedagem e ferramentas de CI/CD ao Git, tornando-o uma plataforma completa de desenvolvimento.

git init
Inicializa um novo repositório Git na pasta atual.
git clone [url]
Copia um repositório remoto inteiro para a máquina local, incluindo todo o histórico.
git status
Exibe o estado dos arquivos: modificados, staged ou não rastreados.
git add .
Adiciona todas as alterações na staging area, preparando-as para o commit.
git commit -m "msg"
Registra o snapshot atual no histórico local com uma mensagem descritiva.
git push origin main
Envia os commits locais para o repositório remoto no GitHub.
git pull
Baixa e integra as atualizações do repositório remoto para a máquina local.
git branch -b feature
Cria e muda para uma nova branch, isolando o desenvolvimento de uma funcionalidade.
git merge [branch]
Une o histórico da branch especificada à branch atual (geralmente main).
04 — GitHub Flow

Fluxo de trabalho

O GitHub Flow é um modelo leve baseado em branches para equipes que fazem deploys frequentes.

01
Criar branch
Cria uma branch isolada a partir da main para desenvolver a nova feature.
02
Commits
Realiza commits frequentes com mensagens claras descrevendo cada mudança.
03
Pull Request
Abre um PR no GitHub para revisão de código pelos colaboradores da equipe.
04
Revisão
Colegas comentam, aprovam ou solicitam ajustes antes de aceitar o código.
05
Merge
Após aprovação, a branch é fundida à main e o código vai para produção.
